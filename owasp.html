Security Guide

OWASP Top 10 — 2021 Edition

The OWASP Top 10 is the globally recognised standard for web application security risks. This guide explains each category in plain language and shows how WebGuard detects it.

WebGuard scans for 7 of 10 OWASP Top 10 categories

What is the OWASP Top 10?

The Open Web Application Security Project (OWASP) is a non-profit foundation dedicated to improving software security. Every few years, OWASP publishes its Top 10 — a consensus list of the most critical security risks facing web applications, based on data from hundreds of organisations worldwide.

The 2021 edition is the current standard. It reorganised and renamed several categories compared to the 2017 list to reflect how the threat landscape has evolved — for example, adding Insecure Design as a distinct category for the first time.

WebGuard's detection algorithms are built and tested against the OWASP Top 10: 2021. Every vulnerability WebGuard finds is mapped to its corresponding OWASP category in reports.
A01
Broken Access Control
Moved up from #5 in 2017 — now the most prevalent risk
Detected by WebGuard

What it is: Access control enforces policy such that users cannot act outside of their intended permissions. When these controls fail or are missing, attackers can access unauthorised functionality or data — viewing other users' accounts, modifying other users' data, or accessing administrative functions.

University example: A student accessing another student's exam results by changing an ID number in the URL. An unauthenticated user reaching the admin dashboard directly via URL.

Vulnerable URL pattern
https://portal.edu.ng/results?student_id=12345
Changing the student_id reveals another student's records if access control is missing.

How to fix: Enforce access control server-side on every request. Never rely solely on client-side checks. Log access failures and alert on repeated failures.

A02
Cryptographic Failures
Previously called "Sensitive Data Exposure"
Detected by WebGuard

What it is: Failures related to cryptography that lead to exposure of sensitive data. This includes transmitting data in clear text, using weak cryptographic algorithms, and improper certificate management.

University example: A student portal served over HTTP instead of HTTPS. Passwords stored using MD5 instead of bcrypt. An expired SSL certificate on the fee payment system.

How WebGuard checks: SSL/TLS certificate validity, expiry date, cipher suite strength, and protocol version. HTTP vs HTTPS detection for all pages handling sensitive data.

How to fix: Enforce HTTPS on all pages. Use TLS 1.2 or higher. Store passwords with bcrypt or Argon2. Never transmit sensitive data in URL parameters.

A03
Injection
Includes SQL injection, XSS, command injection
Detected by WebGuard

What it is: Injection flaws occur when untrusted data is sent to an interpreter as part of a command or query. SQL injection, cross-site scripting (XSS), and OS command injection all fall under this category.

University example: A login form vulnerable to SQL injection where entering ' OR '1'='1 bypasses authentication entirely. An XSS payload in a name field that steals session cookies.

SQL Injection payload (test)
' OR '1'='1' --
If the login accepts this as a valid username, the database query is vulnerable to injection.

How to fix: Use parameterised queries (prepared statements) for all database interactions. Validate and sanitise all user input. For XSS: HTML-encode output and implement Content Security Policy (CSP) headers.

A04
Insecure Design
New category in 2021 — focuses on design-level flaws
Partial coverage

What it is: Insecure design represents missing or ineffective security controls at the design stage. Unlike implementation bugs, these are flaws baked into the architecture — for example, designing a password reset flow that doesn't verify identity before allowing password change.

University example: A "forgot password" flow that emails the actual password in plain text rather than a reset link. A student grade entry system with no audit trail or approval workflow.

How to fix: Use threat modelling during system design. Establish secure design patterns. Conduct security-focused code reviews at the design stage, not only during testing.

A05
Security Misconfiguration
Most commonly found issue in practice
Detected by WebGuard

What it is: Security misconfiguration is the most commonly found issue. It includes missing security headers, default credentials left unchanged, unnecessary services enabled, verbose error messages exposing stack traces, and directory listing enabled on web servers.

How WebGuard checks: HTTP response headers are inspected for the presence of: Strict-Transport-Security, Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, and Referrer-Policy. Missing headers are reported as Medium severity findings.

How to fix: Establish a hardening checklist for server configuration. Remove default accounts. Disable directory listings. Enable all recommended security headers. Review error handling to prevent stack trace disclosure.

A06
Vulnerable and Outdated Components
Third-party libraries and frameworks with known CVEs
Future version

What it is: Components such as libraries, frameworks, and other software modules run with the same privileges as the application. If a vulnerable component is exploited, such an attack can facilitate serious data loss or server takeover.

University example: A university LMS running an outdated version of WordPress or Moodle with a known remote code execution vulnerability. jQuery 1.x serving pages vulnerable to documented XSS exploits.

How to fix: Maintain an inventory of all third-party components. Subscribe to security advisories. Establish a patch management process. Use tools like OWASP Dependency-Check or Snyk.

Component version detection is planned for a future WebGuard release. Current version focuses on runtime vulnerability detection.
A07
Identification and Authentication Failures
Formerly "Broken Authentication"
Detected by WebGuard

What it is: Authentication and session management functions that are implemented incorrectly allow attackers to compromise passwords, keys, or session tokens, or to exploit other implementation flaws to assume other users' identities.

University example: A student portal login with no rate limiting — allowing unlimited password guessing. Session cookies missing the HttpOnly or Secure flags. No CSRF token on the login form.

How WebGuard checks: Analyses login forms for missing CSRF tokens, checks session cookie attributes (HttpOnly, Secure, SameSite), and tests for absence of account lockout or rate limiting mechanisms.

How to fix: Implement rate limiting on authentication endpoints. Require multi-factor authentication for administrative accounts. Set HttpOnly, Secure, and SameSite=Strict on all session cookies. Use CSRF tokens on all state-changing forms.

A08
Software and Data Integrity Failures
New in 2021 — includes insecure CI/CD and auto-updates
Future version

What it is: This category covers failures relating to code and infrastructure that does not protect against integrity violations — for example, loading JavaScript from untrusted CDNs without Subresource Integrity (SRI) checks, or an insecure deserialisation pipeline.

How to fix: Use Subresource Integrity (SRI) hashes when including external scripts. Sign software packages and updates. Review CI/CD pipelines for integrity checks. Avoid deserialising untrusted data.

A09
Security Logging and Monitoring Failures
Critical for breach detection and forensics
Future version

What it is: Without logging and monitoring, breaches cannot be detected. This category covers insufficient logging of security-relevant events, failure to monitor logs for suspicious activity, and absence of incident response procedures.

University example: A student portal with no log of failed login attempts — making brute force attacks invisible. No alerting when an admin account logs in at unusual hours.

How to fix: Log all authentication events (success and failure), access control failures, and input validation failures. Centralise logs in a SIEM. Establish alerting thresholds for anomalous activity.

A10
Server-Side Request Forgery (SSRF)
Added in 2021 from community survey data
Partial coverage

What it is: SSRF flaws occur whenever a web application fetches a remote resource without validating the user-supplied URL. An attacker can coerce the server to send requests to unexpected destinations — including internal network services behind firewalls.

University example: A "check external link" feature on an LMS that can be manipulated to make the server request internal admin interfaces or cloud metadata endpoints.

How to fix: Sanitise and validate all client-supplied URLs. Use an allowlist for permitted domains. Disable HTTP redirections in outbound server requests. Enforce network-layer controls to block server requests to internal resources.
